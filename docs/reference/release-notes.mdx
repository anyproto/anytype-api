A running, **calendar-dated** log of API changes as they ship in `anytype-heart` — including backward-compatible additions that do not introduce a new API version.

:::info How this differs from the [Changelog](./changelog)

The [Changelog](./changelog) is organized by **API version** (the `Anytype-Version` value). A version's section describes what that version's contract looks like, and a new version is minted only for **breaking** changes.

These **Release notes** are organized by **the `anytype-heart` release a change actually shipped in**. Backward-compatible additions become available to the current API version without a new version date, so they are recorded here by ship date rather than backdated into a released version's section.

Each entry notes the API version the change applies to and the `anytype-heart` release that first shipped it. Dates and versions are derived from the served `ApiVersion` history in `anytype-heart` (`core/api/server/middleware.go`) and the first stable release tag containing each change.
:::

## 2026-05-13 — anytype-heart v0.50.5

**API version `2025-11-08` (unchanged).**

### Added

- File management endpoints — a backward-compatible addition; clients on `2025-11-08` gain access without changing the `Anytype-Version` header:
  - `POST /v1/spaces/:space_id/files` to upload a file
  - `GET /v1/spaces/:space_id/files/:file_id` to download a file
  - `DELETE /v1/spaces/:space_id/files/:file_id` to delete a file

  Availability depends on the running `anytype-heart` build. A client can confirm support by checking for the `files` paths in the spec served at `GET /docs/openapi.json` by the local instance.

## 2025-12-11 — anytype-heart v0.46.3

**API version `2025-11-08` introduced** (served-version bump committed 2025-11-09).

Markdown body patching, object type updates, inline tag creation, and property-based filtering. See the [`2025-11-08` Changelog](./changelog#2025-11-08) for the full contract, including changed, fixed, and deprecated items.

### Added

- Markdown body patching via `UpdateObjectRequest`
- Update an object's type by patching `type_key`
- Inline tag creation in `CreatePropertyRequest` for `select` / `multi_select`
- Property-based filtering on listing and search endpoints

## 2025-06-17 — anytype-heart v0.41.0

**API version `2025-05-20` introduced** (served-version bump committed 2025-05-09).

Challenge-based authentication, stable type/property keys, rate limiting, and built-in SVG icons. This version contains **breaking changes** — see the [`2025-05-20` Changelog](./changelog#2025-05-20) for the full list before upgrading a pinned client.

### Added

- New authentication endpoints: `POST /v1/auth/challenges`, `POST /v1/auth/api_keys`
- Stable, caller-specified type and property keys
- Built-in SVG type icons

## 2025-05-06 — anytype-heart v0.40.20

**API version `2025-04-22` introduced** (served-version bump committed 2025-04-20).

First stable release carrying the API versioning header. Full CRUD for spaces, types, properties, and tags. This version contains **breaking changes** — see the [`2025-04-22` Changelog](./changelog#2025-04-22).

### Added

- Space management: `PATCH /spaces/:space_id`
- Full type CRUD: `POST` / `PATCH` / `DELETE /spaces/:space_id/types[/:type_id]`
- Full property CRUD under `/spaces/:space_id/properties`
- Full tag CRUD under `/spaces/:space_id/properties/:property_id/tags`

:::note On API version `2025-03-17`

The served `ApiVersion` constant was first added on 2025-04-06 with the value `2025-03-17`, but it was bumped to `2025-04-22` on 2025-04-20 — before the first release (v0.40.20) that contained it. As a result **`2025-03-17` was never the served version in any stable `anytype-heart` release**. Its documented contract (collections and member management, OpenAPI 3.1 upgrade, the `Anytype-Version` response header) reached users folded into the `2025-04-22` release above. The [`2025-03-17` Changelog](./changelog#2025-03-17) is retained as a record of that contract.
:::
